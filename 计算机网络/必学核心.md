# 一、网络分层
网络分层本质就是把一次完整、极其复杂的网络通信，按功能拆分成好几层，每层只干自己分内的事，分工协作完成数据传输。

### 1. 为什么要分层
把复杂的网络通信拆成独立层级，每层只干自己的事。
- **各司其职**：每层只负责自己的功能，不用关心底层 / 上层细节
- **标准化**：不同厂家的电脑、路由器、交换机可以互通
- **方便排错**：网络故障可以定位到某一层，不用全盘排查
- **易迭代**：修改某一层规则，不影响其他层

### 2. 两大分层模型
- 1. **OSI七层模型（理论）**：物理层 → 数据链路层 → 网络层 → 传输层 → 会话层 → 表示层 → 应用层（从下到上）
- 2. **TCP/IP 五层模型（实际）**：物理层 → 数据链路层 → 网络层 → 传输层 → 应用层（从下到上）
![TCP五层协议](../Images/jsjwl1.png)

### 3. TCP/IP 五层逐层详解（从下往上）
##### 第一层：物理层
- **作用**：传输原始比特流 0、1
- **做什么**：定义网线、光纤、WiFi、5G 怎么传电信号 / 光信号 / 无线电波
- **典型设备**：网线、光纤、集线器、网卡物理接口
- **特点**：只传 0 和 1，不纠错、不识别内容，只管把信号发出去

##### 第二层：数据链路层
- **作用**：组帧、MAC 物理地址寻址、差错校验、局域网内转发
- **关键**：把物理层杂乱的 01 流，打包成数据帧；用 MAC 地址（网卡出厂唯一地址）标识设备
- **典型协议**：以太网、ARP（IP 转 MAC）
- **典型设备**：交换机
- **场景**：同一个家里 / 公司局域网内，设备互相通信靠这一层

##### 第三层：网络层
- **作用**：IP 逻辑地址寻址、路由选路、跨网段通信
- **关键**：给设备分配 IP 地址，帮数据包选择最佳传输路径
- **典型协议**：IP、ICMP（ping 命令就是它）
- **典型设备**：路由器
- **场景**：手机连 WiFi 访问百度，跨运营商、跨城市转发数据靠这一层

##### 第四层：传输层
- **作用**：端到端通信、端口区分、可靠 / 不可靠传输
- **关键**：
   1. 用端口号区分电脑上的软件：80 网页、443 加密网页、微信 / 游戏端口等
   2. 两大核心协议：
        TCP：面向连接、可靠、三次握手、丢包重传（网页、微信、文件下载）
        UDP：无连接、速度快、不保证必达（直播、短视频、游戏）
- **特点**：只负责两个程序之间的数据传输，不管走哪条路由

##### 第五层：应用层
- **作用**：直接给用户 / 应用程序提供网络服务
- 我们能直接感知到的一层，浏览器、APP 都在这一层
- **典型协议**：HTTP/HTTPS、DNS 域名解析、FTP 文件传输
- **场景**：你打开网页、发微信、搜网址，都是应用层在工作

### 4. OSI 七层多出的两层
在传输层 和 应用层之间多了两层，现实互联网不用，只是理论：
- **会话层 **：建立、保持、断开通信会话（比如保持登录在线）
- **表示层 **：数据加密、解密、格式转换、编码解析

### 5. 数据传输核心流程：数据封装 & 解封装
- **发送方**（从上往下，每层加头部标签）
应用数据 → 传输层加端口头 → 网络层加 IP 头 → 数据链路层加 MAC 头 → 转为 01 比特流 → 物理层传输
- **接收方**（从下往上，逐层拆标签）
物理层收 01 → 拆数据链路层帧 → 拆网络层 IP 包 → 拆传输层端口 → 最终拿到应用数据
> 一句话总结：下层为上层提供服务，每层只和对等层对话，互不干扰。

# 二、TCP 三次握手、四次挥手
TCP 是**可靠面向连接**的协议，三次握手 <=> 客户端和服务器 互相确认双方「能发、能收」，正式建立通信通道。
### 三次握手
##### 1. 握手流程
1. 第一次握手：客户端 → 服务端（客户端发送 SYN 同步报文）
>意思：我想跟你建立连接，我的初始序号是 X
>作用：服务器知道**客户端能发数据**

2. 第二次握手：服务端 → 客户端（服务端返回 SYN + ACK 报文）
>意思：我收到你的请求了，我也同意连接，我的初始序号是 Y
>作用：客户端知道**服务端能收、也能发数据**

3. 第三次握手：客户端 → 服务端（客户端发送 ACK 确认报文）
>意思：我收到你的同意了，连接正式建立
>作用：服务端知道**客户端能收数据**

##### 2. 握手目的与结果
验证客户端能发、服务端能收，能发、客户端能手 → 全部正常，才敢传数据，保证可靠。

##### 3. 为什么不能两次握手？
如果只两次：服务器收到客户端请求，直接就建立连接。
导致问题：网络延迟，旧的失效请求包迟到发到服务器，服务器依然新建一条无效连接，白白占用资源，导致服务器崩溃。
→ **三次握手 就是为了防止失效报文建立无效连接。**

### 四次挥手
##### 1. 挥手流程
1. 第一次挥手：客户端 → 服务端（客户端发 FIN）
>意思：我没数据要发了，我要关闭发送通道了

2. 第二次挥手：服务端 → 客户端（服务端回 ACK）
>意思：我收到你要关闭的请求了
>**注意：服务端此时还没关！还可以继续给客户端发剩余数据**

3. 第三次挥手：服务端 → 客户端（服务端发 FIN）
>意思：我也没数据了，我也要关闭了

4. 第四次挥手：客户端 → 服务端（客户端回 ACK）
>意思：我知道你也要关了，你可以彻底关闭了

> 什么是报文：电脑之间不能随便发零散的 0 和 1，必须打包成一个规整的包裹，这个打包好的完整数据包，就叫报文。

# 三、DNS 域名解析（输入网址第一步）
### 本质
DNS = 互联网通讯录 → 把域名翻译成 IP 地址，电脑才能找到对方服务器。

### 为什么不能直接用 IP？
1. 数字 IP 太难记，域名好记
2. 网站换服务器、换 IP 时，用户不用改网址，只改 DNS 解析就行
3. 一个域名可以对应多个 IP，用来分流、抗卡顿

### DNS完整解析流程
首先，浏览器输入 www.baidu.com：
1. 浏览器自身缓存
先看自己之前有没有查过百度，有保存 IP → 直接用，结束。

2. 电脑本地 hosts 文件
浏览器没缓存，就看电脑自带的本地通讯录，可以手动自己绑定域名和 IP。

3. 本地 DNS 服务器
本地都没有，就去找运营商 DNS / 路由器 DNS（宽带、手机 WiFi 自动分配的）。

4. 根 DNS 服务器
本地 DNS 不知道，就去问全球根服务器。根服务器不存具体 IP，只告诉你：.com 的域名去找顶级域名服务器。

5. 顶级域名服务器
负责管理 .com .cn .net 这类后缀，它再告诉你：百度的域名去找百度权威服务器。
> 网址最后面的后缀，就是顶级域名：baidu.com 里的 .com、baidu.cn 里的 .cn每一种后缀，都有专门一批顶级域名服务器来管。

6. 权威 DNS 服务器
网站自己的 DNS，直接给出百度真实 IP。

7. 返回
原路把 IP 传回你的浏览器，浏览器拿着 IP 去访问百度服务器。

### 两个关键概念
1. 递归查询：浏览器 → 本地 DNS（我只问一次，你帮我查到底，直接给我最终 IP 结果。）
2. 迭代查询：本地 DNS → 根 → 顶级 → 权威（我不帮你查到底，我只告诉你下一个该找谁，你自己接着问。）

### 补充
1. DNS 默认用 UDP 53 端口，速度快
2. 所有解析结果都会缓存，下次不用重新查，上网更快
3. 劫持 DNS：把你域名解析到错误 IP，就是跳转到诈骗网站

# 四、HTTP 协议
### 定义
HTTP 是互联网应用层的通信规则，专门规定：浏览器 / APP 怎么向服务器发请求、服务器怎么返回数据。
> 通俗比喻->你去奶茶店点单，必须按规矩说：
> 你：我要一杯珍珠奶茶（客户端请求）
> 店员：给你做好的奶茶（服务器响应）
> HTTP 就是这套统一说话规矩，全世界浏览器、服务器都按这个标准交流。

### HTTP 三大核心特点
1. **无连接**：一次请求、一次响应完，立刻断开连接，不一直占用通道。
> 浏览器发 HTTP 请求：我要网页。服务器返回数据：给你网页内容 -> 把 TCP 连接关掉，关闭通道

2. **无状态**：服务器不记住你是谁，每次请求都当陌生人。所以才有 Cookie、Session 来记录登录状态。
3. **明文传**：输数据直接裸奔传输，不加密，容易被窃听、篡改，不安全。所以衍生出 HTTPS（HTTP + 加密）。

### HTTP 工作位置
工作在 TCP/IP 五层模型 —— **应用层**
底层依赖：TCP 传输层 先三次握手建连接，再传 HTTP 数据。

### HTTP 通信流程
1. 浏览器输入网址 → DNS 解析出服务器 IP
2. 浏览器和服务器 TCP 三次握手 建立连接
3. 浏览器发 HTTP 请求
4. 服务器处理业务，返回 HTTP 响应
5. 数据传输完，四次挥手 断开连接

### HTTP 两大核心报文
1. **请求报文（浏览器→服务器）**
三部分：
- 请求行：请求方法 + 网址 + 协议版本
- 请求头：浏览器信息、设备、Cookie、权限等
- 请求体：提交的参数（登录账号、表单数据）

2. **响应报文（服务器→浏览器）**
三部分：
- 状态行：协议版本 + 状态码
- 响应头：服务器信息、缓存规则、编码
- 响应体：真正数据（HTML 网页、JSON 接口、图片、视频）

### HTTP 常用请求方法
- **GET**：查询数据，参数拼在网址上，可缓存、不安全
- **POST**：提交数据（登录、注册），参数放请求体，不缓存
- **PUT**：修改服务器数据
- **DELETE**：删除服务器数据
- **HEAD**：只拿响应头，不要页面内容

### HTTP 常见状态码
- **200**：请求成功，一切正常
- **301**：永久重定向（网址永久搬家）
- **302**：临时重定向
- **404**：页面 / 接口不存在
- **403**：权限不足，禁止访问
- **500**：服务器内部代码报错
- **502**：网关错误，服务器崩了 / 接口不通

### HTTP 缺点
HTTP 是明文传输，容易被窃听、篡改、冒充。所以衍生出 HTTPS，HTTPS 是加密传输，安全。

# 五、HTTPS 协议
### 本质
HTTPS = HTTP + SSL/TLS 安全加密层。就是加了锁的 HTTP，解决 HTTP 明文裸奔、不安全的问题。

### 作用
- 1. **加密**：传输的数据变成密文，中间人看不懂
- 2. **防篡改**：数据中途被改一点，接收方立刻能检测出来
- 3. **身份认证**：确认你访问的是真实官网，不是假冒钓鱼网站

> 浏览器地址栏小锁图标，就是 HTTPS 生效标志。

### SSL/TLS
介于应用层（HTTP）和传输层（TCP） 中间的加密安全层。
> SSL/TLS 不是独立层级，干的是理论网络分层中的表示层和会话层的工作，存在于实际中的应用层

相当于：给 HTTP 套了一层加密防护服。
现在主流用 TLS1.2 / TLS1.3，SSL 已经淘汰，日常统称 SSL/TLS。
